# Plan — РоВ-глубина генерации сценариев

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development или superpowers:executing-plans, задача-за-задачей. Шаги в формате чекбоксов (`- [ ]`).

**Goal:** Довести сгенерированные сценарии до уровня методичек «Разговоры о важном» (вариант 1s): развёрнутый диалоговый ход (несколько реплик «Учитель: …» подряд с конкретным содержанием), явные формируемые ценности и основные смыслы, разноуровневые развёрнутые вопросы ученикам. Целевой объём — кратно больше текущего (~3 КБ → ближе к РоВ ~15–18 КБ).

**Ключевой вывод прототипа (2026-05-21):** GigaChat в ОДИН запрос «насыщает» ~800 токенов и не реагирует на инструкции «не экономь текст», `max_tokens`, ни на подачу реального РоВ-чанка (RAG поднимает тематичность, но не плотность). Значит глубина достигается только **по-этапной генерацией**: отдельный фокусный вызов на каждый этап (узкая задача → модель пишет полно) с RAG-опорой на этот этап, затем сборка. Объём масштабируется числом этапов.

**Architecture:** skeleton (title + goals + **values** + **coreMeanings** + список этапов) → для КАЖДОГО этапа отдельный chat-вызов `buildStageDetailsMessages(...)` с RAG-чанками, релевантными (тема + вид этапа) → сборка в `ScenarioContent` → нормализация типов (`coerceContentTypes`) + хронометража → zod-валидация. Всё через инъекцию зависимостей (как `streamScenario(input, deps)`), юнит-тесты без сети.

**Tech Stack:** Next.js 15, TypeScript, Drizzle (Postgres+pgvector), Vitest, GigaChat (SSE), zod.

**Базис:** ветка `claude/friendly-jackson-731786` (= master). Конвенции CLAUDE.md: один коммит на задачу; TDD для чистой логики; гейты зелёные перед коммитом (`pnpm test`, `pnpm lint`, `tsc --noEmit`, `pnpm build`); UI на русском; juнит-тесты не ходят в сеть/БД.

## Риски
- **Стоимость/латентность:** N вызовов вместо 1-2 (на каждый этап). Rate-limit генерации сейчас считает 1 событие — оставить так (один пользовательский запрос), но следить за временем стрима. Возможный кап на число этапов.
- **Бюджет токенов:** суммарный ответ большой; per-stage снимает обрезание, но общий объём БД-jsonb растёт (приемлемо).
- **Прод-миграция:** если в задаче 1 всё же понадобится колонка — `pnpm db:migrate` на проде (как с 0011).
- **UX стрима:** per-stage → больше промежуточных событий; прогресс-бар уже фазовый, адаптировать.
