# Дизайн — РоВ-уровень генерации через per-block пайплайн

**Дата:** 2026-05-23
**Статус:** утверждён
**Контекст:** backlog #21 («РоВ-глубина генерации»). Предыдущая фаза (`2026-05-21-plan-rov-depth-generation.md`) ввела per-stage генерацию + ценности/смыслы и подняла объём ~3 КБ → ~5 КБ, но целевой уровень РоВ (~15-18 КБ) не достигнут. Эта спека — следующий шаг к целевому качеству.

## Проблема

Цель — довести сгенерированные сценарии до уровня методичек «Разговоры о важном» (вариант `1s`): развёрнутый многоходовой ход «Учитель: …» с конкретикой, явные ценности и смыслы, разноуровневые вопросы, связность этапов. Пользователь явно подтвердил: «качество как у РоВ», поднимаем всё (глубина хода, полнота структуры, связность, содержательность).

**Замер реального РоВ (живой RAG-корпус, 2026-05-23):** один сценарий `1s` = ~18-20 КБ, разбит на ~9-11 плотных блоков по ~800-1000 токенов каждый. Блок = длинный монолог «Учитель: …» с фактами/историями/отсылками + блоки вопросов / таблицы / командные задания.

**Корневая причина текущего потолка:** GigaChat «насыщает» ~800 токенов на один вызов независимо от инструкций «не экономь» / `max_tokens` (подтверждено прототипом 2026-05-21). Текущий пайплайн делает **1 вызов на этап** (3 этапа) → ~800 ток. на весь этап → ~5 КБ. РоВ = ~10 блоков. Значит объём достижим только дроблением до **1 вызова на блок (активность)**, тогда суммарный объём масштабируется числом блоков.

## Решения (зафиксированы с пользователем)

1. **Подход A** — per-activity (per-block) генерация + контент-план в каркасе + катящийся контекст. (Отклонены: B per-stage — не пробивает потолок ~5-7 КБ; C с LLM-ревизией — прототип показал, что модель игнорирует «расширь».)
2. **Латентность не важна, приоритет качества** — допустимо ~2 мин на генерацию (1 каркас + ~9 блоков последовательно).
3. **Гейт качества — детерминированный per-block** (локальные пороги + авто-перегенерация тонкого блока). Без LLM-судьи.

## Архитектура

```
вход (форма)
  → RAG retrieve (topK ↑ для большего фактажа) + prematch (как сейчас, best-effort, не валит генерацию)
  → ФАЗА 1: skeleton — 1 вызов, стримится
       Каркас несёт КОНТЕНТ-ПЛАН: в каждом этапе список «брифов блоков» { type, focus },
       где focus описывает, ЧТО раскрывает блок и какой смысл (coreMeaning) несёт.
       main: 2-4 брифа; engage/reflection: 1-2. Всего ~8-11 блоков.
       Смыслы распределяются по блокам на этапе плана → повторов нет by design.
  → ФАЗА 2: цикл ПО БЛОКАМ (последовательно, не по этапам):
       для каждого брифа b этапа s:
         msgs = buildBlockMessages(input, skeleton, s, b, ragChunksForBlock, runningContext)
         block = generateValidated(chat, msgs, parseBlock, { attempts, corrective })
         while !qualityGate(block, b) && retries < MAX_BLOCK_RETRIES:
             block = generateValidated(... corrective="блок тонкий: добавь реплики/факт/вопросы")
         (если так и тонкий — берём лучшую попытку, не валим; помечаем в meta)
         runningContext.push(сводка block)   # ~200 симв. текста блока
         emit { type:'block', index, total }
  → сборка ScenarioContent → coerceContentTypes → normalizeChronometry
  → ФИНАЛЬНЫЙ сводный чек (локально): объём, нет дублей этапов, смыслы покрыты → warnings в meta
  → save
```

Всё через инъекцию зависимостей (как `streamScenario(input, deps)`) — юнит-тесты без сети/БД.

## Компоненты

### `lib/scenario/schema.ts`
- `skeletonSchema` расширяется: каждый этап получает `blocks: [{ type, focus }]` (бриф блока). `type` — из enum активностей; `focus` — строка-описание.
- Новый `blockSchema` = одиночная активность (`type`, `text`, `questions?`) — для валидации ответа одного блока. (`stageActivitiesSchema` из прошлой фазы удаляется — осиротеет.)
- `ScenarioContent` (финальный) НЕ меняется: `blocks` каркаса — промежуточный план, в БД сохраняется развёрнутый `stages[].activities[]`.

### `lib/scenario/prompt.ts` (`PROMPT_VERSION → v6-rov-blocks-2026-05-23`)
- `buildSkeletonMessages` — теперь требует контент-план `blocks` на каждый этап (что раскрывает каждый блок).
- Новый `buildBlockMessages(input, skeleton, stage, brief, ragChunks, runningContext)` — генерит ОДИН плотный блок. System давит на плотность одного блока (несколько реплик «Учитель: …», конкретика, разноуровневые вопросы). User даёт: бриф + связанный смысл + RAG-чанки блока + runningContext («уже раскрыто — не повторяйся, развивай»).
- Удаляются осиротевшие `buildStageDetailsMessages`, `buildDetailsMessages` (и `buildMessages`, если не используется вне стрима — проверить перед удалением).

### `lib/scenario/context.ts` (новый, чистый, TDD)
- `buildRunningContext(blocks: GeneratedBlock[]): string` — программно (0 LLM-вызовов) собирает сводку готовых блоков: по строке `Этап «X» (тип): <первые ~200 симв. текста>`. Прокидывается в `buildBlockMessages`.

### `lib/scenario/quality.ts` (новый, чистый, TDD)
- `checkBlock(block, brief): { ok: boolean; reasons: string[] }` — детерминированные пороги:
  - длина `text` ≥ `MIN_BLOCK_CHARS` (env, дефолт 600);
  - число вхождений `Учитель:` ≥ 2 для `engage`/`main` (многоходовость);
  - для `type==='discussion'` — `questions.length ≥ 3`.
- `checkScenario(content): { warnings: string[] }` — финальный сводный чек: общий объём ниже ожидаемого, дубли заголовков этапов, непокрытые `coreMeanings`. Только warnings (не блокирует save).
- Пороги/лимиты через env: `MIN_BLOCK_CHARS`, `MAX_BLOCK_RETRIES` (дефолт 2).

### `lib/scenario/stream.ts`
- Цикл переписывается с per-stage на per-block: после каркаса разворачиваем `stages[].blocks[]` в плоскую очередь блоков с привязкой к этапу; на каждый блок — фокусный вызов + гейт + ретраи; собираем `stages[].activities[]`.
- `StreamEvent` дополняется `{ type:'block'; index: number; total: number }` (после каркаса известно `total`).
- `chunksForStage` (из прошлой фазы) переиспользуется для маршрутизации чанков на блок по `type`/`kind`.
- `meta` дополняется: число тонких блоков, не добравших порог; warnings финального чека.

### `lib/scenario/regenerate.ts`
- Точечная 🎲-регенерация одной активности переводится на `buildBlockMessages` для единообразия плотности (опционально, если не раздувает scope — иначе оставить как есть, отметить в плане).

### `components/generation/GenerationStream.tsx`
- Прогресс per-block: «Блок i из N» по событиям `{ type:'block' }`. Каркас по-прежнему стримится для раннего отклика.

### `scripts/seed-shared.ts`
- Пере-сид библиотеки новым пайплайном (как в прошлой фазе) — код-часть; живой прогон на проде — ручной шаг после деплоя.

## Поток данных

Форма → `streamScenario(input, deps)` → SSE-события (`phase` → `skeleton` → N×`block` → `done`/`error`) → клиент. Сохранение через `deps.save` (insert `scenarios`+`scenario_versions`+`generations`+`embedding`, изоляция по `user_id` — без изменений).

## Обработка ошибок

- RAG/prematch — best-effort, сбой не валит генерацию (как сейчас).
- Каркас невалиден → repair-pass (как сейчас) → если нет, `error`.
- Блок невалиден по zod → repair-pass; тонкий по гейту → ретрай; не добрал после ретраев → берём лучшую попытку, не валим (помечаем в meta).
- Сборка не прошла zod → `error`.

## Тестирование

- TDD (чистая логика, без сети/БД): `quality.ts` (`checkBlock`/`checkScenario`), `context.ts` (`buildRunningContext`), новые промпт-билдеры (структура messages, наличие runningContext/брифа), парс одиночного блока + coerce.
- `streamScenario` — юнит-тест с инъекцией `chat`/`retrieve`/`prematch`/`save` (стаб), проверка: per-block цикл, гейт-ретрай тонкого блока, сборка, события `block`.
- Гейты перед каждым коммитом: `pnpm test`, `pnpm lint`, `tsc --noEmit`, `pnpm build`.
- Живой UAT (ручной, перед демо): прогон против GigaChat, замер объёма (цель ~15-18 КБ), визуальная проверка многоходовости/связности/вопросов.

## Риски

- **Латентность ~2 мин** — принято (приоритет качества); прогресс-бар per-block держит UX.
- **Rate-limit** считает 1 событие на пользовательскую генерацию — оставляем (внутренние N вызовов не штрафуем).
- **Объём jsonb** растёт (~18 КБ/сценарий) — приемлемо.
- **Фактологичность** опирается на RAG-чанки РоВ; гейт проверяет ТОЛЬКО плотность, не достоверность фактов — осознанно вне scope (риск галлюцинаций остаётся).
- **Без миграций** (контент в jsonb; `blocks` каркаса — промежуточный план).

## Вне scope

- LLM-судья качества; фаза ревизии/дораздувания.
- РоВ-специфика без основы: отсылки к слайдам презентации, встраивание видео/интерактива, генерация презентаций.
- Проверка достоверности фактов.
- Шаблоны «Орлята России»/РДДМ.
