# Дизайн — Регенерация активности на per-block пайплайне (#22 + #31)

**Дата:** 2026-05-23
**Статус:** утверждён
**Контекст:** backlog #22 (🎲-регенерация на старом промпте) + #31 (введение/рефлексия превращаются в игру). Один корень: `lib/scenario/regenerate.ts` использует устаревший `buildActivityMessages` — без роли этапа, без катящегося контекста, без гейта качества. После перехода генерации на per-block (`v6-rov-blocks`) точечная регенерация отстала: блоки выходят мельче остального сценария и игнорируют роль этапа.

## Проблема

1. **#22:** Регенерированная активность проседает по глубине/связности относительно остального РоВ-сценария (старый промпт не несёт роль этапа, основные смыслы, контекст соседних блоков, не прогоняется через `checkBlock`).
2. **#31:** При регенерации введения (engage) и рефлексии (reflection) тип меняется на `game`/`task` — рвёт педагогическую логику. Реальный пример: в «Безопасный интернет» регенерация введения дала игру «Опасности онлайн» с карточками вместо вводной беседы.

## Решения (зафиксированы с пользователем)

1. **Тип выбирает учитель** перед регенерацией (UI-селектор у 🎲, дефолт = текущий тип блока). Итоговому блоку тип проставляется по выбору жёстко (не доверяем модели) — гарантия против «игры во введении».
2. **Гейт качества как в генерации:** регенерированный блок проходит `checkBlock` и при «тонкости» перегенерируется до `MAX_BLOCK_RETRIES` (как per-block цикл в `stream.ts`).

## Архитектура

Переписать регенерацию на тот же per-block промпт и гейт, что и генерация; вынести общий шаг «сгенерировать один блок с гейтом» в переиспользуемый хелпер.

```
editor 🎲 (+ выбор типа)
  → regenerateActivityAction(scenarioId, stageIndex, activityIndex, type)
      — грузит ScenarioContent с изоляцией user_id (как сейчас)
      — RAG retrieve best-effort (как сейчас)
      — reconstructSkeleton(content, stage)         # title, goals, coreMeanings, нужный stage
      — brief = { type: выбранный, focus: stage.title }
      — runningContext = buildRunningContext(все активности КРОМЕ регенерируемой)
      — regenerateActivity(...) → generateBlockWithGate(...)
      — activity.type = выбранный тип (форсим)
      — лог в generations (как сейчас)
  → редактор подставляет новый блок (dirty, явный Save — как сейчас)
```

## Компоненты

### `lib/scenario/block-gen.ts` (новый, чистый) — общий хелпер
- `generateBlockWithGate(chat, messages, stageKind, opts): Promise<{ value: Activity; repaired: boolean; accepted: boolean } | null>` — инкапсулирует: `generateValidated(chat, msgs, parseBlock, ...)` → `checkBlock(value, stageKind)` → при провале добавляет корректирующие реплики и повторяет до `maxRetries` (деф. `MAX_BLOCK_RETRIES`). Возвращает лучший результат + флаги. Использует существующие `parseBlock`-логику (вынести `parseBlock` сюда же или импортировать).
- Извлекается из текущего инлайн-цикла `stream.ts` (строки per-block ретраев). `stream.ts` начинает звать этот хелпер — поведение генерации не меняется (тот же гейт/ретраи).
- TDD: тонкий блок → ретрай → принят; невалидный → repair; «потолок» ретраев → лучший результат с `accepted:false`.

### `lib/scenario/regenerate.ts` (переписать)
- Удалить `buildActivityMessages` + `ACTIVITY_SCHEMA_HINT` (осиротеют — проверить, что больше нигде не используются).
- `RegenerateArgs` расширяется: помимо текущих полей нужны `content: ScenarioContent` (или собранный `skeleton`-вид) и `targetType: string` и `runningContext: string`. Уточнить точную форму в плане, минимально достаточную для `buildBlockMessages`.
- `regenerateActivity(args, deps)` строит `buildBlockMessages(input, skeleton, stage, brief, ragChunks, runningContext)` → `generateBlockWithGate(chat, msgs, stage.kind, ...)` → форсит `value.type = targetType` → возвращает `Activity`.
- DI `chat` сохраняется (юнит-тесты без сети).

### `app/app/scenarios/[id]/actions.ts`
- `regenerateActivityAction` принимает новый параметр `type` (валидируется как один из enum активностей). Грузит сценарий (изоляция `user_id`), reconstructs skeleton из `content`, собирает `runningContext` из всех активностей кроме целевой, зовёт `regenerateActivity`. RAG retrieve, лог в `generations`, rate-limit `MAX_REGEN_PER_DAY` — без изменений.

### `components/.../editor.tsx`
- У 🎲-контрола per-activity — компактный `<select>` типа (5 значений из `options`), дефолт = текущий `activity.type`. Значение передаётся в экшен. UI на русском, в стиле существующих контролов.

## Поток данных
Без изменений в БД и контракте `ScenarioContent`. Регенерация заменяет ОДНУ активность в клиентском состоянии редактора; сохранение — прежним явным Save (`saveScenarioAction`).

## Обработка ошибок
- RAG retrieve best-effort (сбой не валит регенерацию) — как сейчас.
- `generateBlockWithGate` вернул `null` (все попытки невалидны) → экшен возвращает ошибку, редактор показывает сообщение (как сейчас на провал regen).
- «Тонкий» после ретраев → берём лучший результат (не валим) — как в генерации.

## Риски
- **Латентность регенерации:** теперь до ~3 вызовов (гейт-ретраи) вместо 1-2. Принято (приоритет качества); rate-limit `MAX_REGEN_PER_DAY` уже ограничивает.
- **Вынос хелпера из `stream.ts`:** риск регрессии генерации — снимается тем, что `stream.test.ts` остаётся зелёным и хелпер инкапсулирует ровно текущую логику.
- **`checkBlock` для game/quiz в main:** правило «≥2 реплик Учитель:» применяется к engage/main независимо от типа (существующее поведение генерации) — для чистой игры может вызвать лишние ретраи. Не новый риск; оставляем как есть.
