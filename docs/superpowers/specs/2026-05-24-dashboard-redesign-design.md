## Цель

Сделать главную страницу приложения полезной за рамками «последних 10 сценариев»:
1. **Поиск/фильтр по личной библиотеке** (#28) — за учебный год сценариев много, нужно «найти тот про дружбу».
2. **Виджет «Ближайшие мероприятия»** — 3 ближайших события по дате: из загруженного плана воспитательной работы либо (при отсутствии плана) из статического календаря поводов.

Без миграций. Изоляция по `user_id` на всех запросах к пользовательским данным.

## Архитектура и поток данных

`app/app/page.tsx` остаётся **серверным компонентом** (RSC, async). Читает `searchParams`:
`?q=` (текст), `?direction=`, `?grade=`, `?format=` (фильтры).

- **Поисковая строка + фильтры** — клиентский компонент `components/dashboard/ScenarioSearch.tsx` (`'use client'`). Поле ввода + чипы/селекты фильтров пушат значения в URL через `useRouter().replace(...)` + `useSearchParams()`. Debounce ~300 мс на текстовом вводе, чтобы не дёргать навигацию на каждый символ. Сам список сценариев рендерит сервер по обновлённым `searchParams`.
- **Виджет «Ближайшие мероприятия»** — чистый серверный рендер (без интерактива). Данные считает чистая функция (см. ниже), страница только маппит результат в карточки-ссылки.

## Компонент 1 — Поиск по своим сценариям (#28)

### Запрос
В `app/app/page.tsx`:
```
WHERE scenarios.user_id = :userId
  AND (:q IS NULL OR title ILIKE '%' || :q || '%')
  AND (:direction IS NULL OR direction = :direction)
  AND (:grade IS NULL OR grade = :grade)
  AND (:format IS NULL OR format = :format)
ORDER BY created_at DESC
LIMIT 100
```
- Текстовый матч — `ILIKE` по `title` (регистронезависимо). **YAGNI:** без поиска по телу сценария — название покрывает основной сценарий «найти тот про дружбу».
- Фильтры строятся из `lib/scenario/options.ts`: `DIRECTIONS`, `FORMATS`, классы (1–11 + СПО=12 через `formatGrade`).
- Невалидные значения фильтров (не из справочника) игнорируются (трактуются как «без фильтра») — защита от мусора в query.
- Изоляция по `user_id` — обязательный предикат во `WHERE`, не опускается ни при каком сочетании фильтров.

### Поведение списка
- **Единый список заменяет прежние «последние 10» полностью.** При пустом запросе и без фильтров — все сценарии пользователя, `ORDER BY created_at DESC`, cap 100.
- Карточки сценариев — как сейчас (title + бейджи direction/grade/format), ссылка на `/app/scenarios/{id}`.
- Пустой результат при активном запросе → сообщение «Ничего не найдено» (отличать от «совсем пусто» — приглашение создать первый сценарий).

## Компонент 2 — Виджет «Ближайшие мероприятия»

### Чистая логика: `lib/dashboard/upcoming.ts` (TDD)
Функция без БД- и сетевых зависимостей. Сигнатура (ориентировочно):
```ts
type UpcomingItem = {
  title: string
  date: Date            // конкретная разрешённая дата (для сортировки/показа)
  source: 'plan' | 'calendar'
  planTopicId?: string  // для тем плана
  scenarioId?: string   // если у темы плана уже есть сценарий
  calendarDate?: string // 'MM-DD' для повода календаря
}

function pickUpcoming(args: {
  today: Date
  planTopics: { id: string; title: string; plannedDate: string | null; scenarioId: string | null }[]
  calendar: CalendarOccasion[]
}): UpcomingItem[]   // не более 3
```

**Правила:**
1. **Парсинг `plannedDate`** (free-text `DD.MM` или `DD.MM.YYYY`, разделитель `.` или `/`):
   - С явным годом → конкретная дата; если она в прошлом относительно `today` — тема не попадает в «ближайшие».
   - Без года → привязываем к **ближайшему будущему вхождению** (как повторяющийся повод): если `DD.MM` в этом календарном году уже прошло — берём следующий год.
   - Неразборчивая/невалидная дата (например 31.02 или мусор) → тема пропускается.
2. Если среди тем плана есть ≥1 валидная будущая дата → берём **3 ближайшие** темы плана (по возрастанию даты). Это приоритетный источник.
3. Если у пользователя нет тем плана с датами **или** ни одна не парсится в будущую дату → **fallback на календарь**: 3 ближайших повода из `CALENDAR_EVENTS`. Поводы — `MM-DD`, повторяющиеся ежегодно: разрешаем в ближайшее будущее вхождение тем же правилом «без года».
4. Источники не смешиваем: либо план, либо календарь (по приоритету выше).

### Ссылки (строит страница, не чистая функция)
- Тема плана, у которой уже есть сценарий (`scenarioId` != null) → ссылка на `/app/scenarios/{scenarioId}`.
- Тема плана без сценария → `/app/new?topic={title}&planTopicId={id}` (существующий префил-поток).
- Повод календаря → `/app/new?topic={title}&calendarDate={MM-DD}` (существующий префил-поток).

### Данные на странице (изоляция по user_id)
- Темы плана: `SELECT id, title, planned_date, ... FROM plan_topics WHERE user_id = :userId` (+ join/lookup существующего сценария темы через `scenarios.source_plan_topic_id`, тоже под `user_id`). Переиспользуем уже считаемое на странице покрытие тем (`coveredIds`), расширив его до `topicId → scenarioId`.
- Календарь: статический `CALENDAR_EVENTS`.

2. **Виджет «Ближайшие мероприятия»** — 3 карточки-ссылки (заголовок секции + подпись источника «из вашего плана» / «памятные даты»).
3. Карточки прогресса по планам (`planStats`) — как сейчас.
4. Карточка «Библиотека сообщества: N» — как сейчас.

## Обработка ошибок / краевые случаи
- Нет сценариев вообще → текущее приглашение «Пока пусто».
- Поиск дал 0 при активном запросе → «Ничего не найдено».
- Нет ни плана, ни валидных дат, и календарь пуст (теоретически) → виджет не рендерится.
- `plannedDate = null` или неразборчив → тема молча выпадает из виджета (но остаётся в прогрессе плана).

## Тестирование
- **TDD** для `lib/dashboard/upcoming.ts`:
  - парсинг `DD.MM` без года → ближайшее будущее вхождение (включая переход через год);
  - `DD.MM.YYYY` в прошлом → исключается; в будущем → включается;
  - выбор ровно 3 ближайших по возрастанию даты;
  - fallback план→календарь при отсутствии валидных дат плана;
  - неразборчивые даты пропускаются;
  - пустые входы → пустой результат.
