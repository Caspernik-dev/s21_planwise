# Соответствие сценариев ФГОС/ФОП — P1+P2

> Дата: 2026-05-30
> Контекст: `docs/fgos-fop-compliance.md` фиксирует, что «соответствие ФГОС» — это
> отражение в сценарии четырёх сущностей: 8 направлений воспитания, личностных
> результатов по уровню, трёхчастной структуры, санитарных лимитов. У продукта
> уже есть направления, структура, нормализация хронометража и календарь поводов.
> Главный пробел — **личностные результаты** (нет в `ScenarioContent`) и
> **методическая шапка экспорта** (есть только базовая). Дополнительно — нужен
> возрастной потолок длительности и мягкая проверка наличия рефлексии.

## 1. Цели

1. Каждый новый сценарий содержит блок **«Планируемые личностные результаты»**
   — 3–5 дословных формулировок из текста ФГОС, привязанных к уровню
   образования (НОО/ООО/СОО) и направлению воспитания.
2. PDF/DOCX-экспорт открывается с полноценной **методической шапкой**
   (тема, направление, класс/уровень, длительность, формат, цель, ценности,
   оборудование) и блоком личностных результатов.
3. Сервер и форма генерации не пропускают **превышение санитарного лимита**
   длительности занятия по возрасту (1 класс ≤ 35 мин; 2–11, СПО ≤ 45 мин).
4. Сценарий получает мягкий warning, если в нём нет **этапа рефлексии** или в
   рефлексии нет вопросов для обратной связи.

## 2. Что НЕ входит

- Маркетинг лендинга (P3 из исследования) — отдельным микропатчем, без спеки.
- Регенерация личностных результатов отдельной кнопкой 🎲.
- Загрузка PDF приказов ФГОС в RAG-корпус.
- Backfill старых сценариев (поле `personalResults` остаётся пустым в БД для
  ранее созданных записей; UI/экспорт обрабатывают отсутствие как «—»).
- UI выбора уровня образования вручную (всегда выводится из `grade`).
- Изменения в `seed-shared.ts` или принудительная регенерация библиотеки
  сообщества. Опциональный пере-сид — после деплоя, вручную.
- Жёсткая валидация: учителю в редакторе разрешено править формулировки
  личностных результатов руками (whitelist применяется только к выходу LLM).

## 3. Архитектура

### 3.1. Новые файлы

- `lib/scenario/levels.ts`
  - `type Level = 'NOO' | 'OOO' | 'SOO'`
  - `gradeToLevel(grade: number): Level` — 1–4→НОО, 5–9→ООО, 10–11→СОО, 12 (СПО) →СОО.
  - `type CanonicalDirection` — 8 канонических направлений ФГОС.
  - `canonicalDirection(direction: Direction): CanonicalDirection` — маппинг 11→8.
  - `levelLabel(level: Level): string` — «НОО»/«ООО»/«СОО» для UI.

- `lib/scenario/personal-results.ts`
  - Статический `CATALOG: Record<Level, Record<CanonicalDirection, string[]>>`
    — 24 ячейки, по 5–8 дословных формулировок из ФГОС НОО/ООО/СОО
    (приказы № 286, 287, 413+732).
  - `getCatalog(level: Level, direction: Direction): string[]`

    — нормализует пробелы (`\s+` → ` `, trim), возвращает только те элементы
    входа, что точно совпадают с какой-либо строкой каталога.

    — обёртка: валидирует, если осталось < 3 — добирает первыми из каталога

## 4. Каталог личностных результатов — источник и формат

**Источник.** Дословные формулировки выписываются из текстов ФГОС:
- ФГОС НОО — приказ Минпросвещения № 286 от 31.05.2021, раздел III п. 31
  «Личностные результаты освоения программы НОО».
- ФГОС ООО — приказ № 287 от 31.05.2021, раздел III п. 41
  «Личностные результаты освоения программы ООО».
- ФГОС СОО — приказ № 413 от 17.05.2012 (ред. приказа № 732 от 12.08.2022),
  раздел III «Личностные результаты».

В каждом ФГОС личностные результаты сгруппированы по направлениям
(«гражданское и патриотическое воспитание», «духовно-нравственное», «эстетическое»,
«физическое воспитание и формирование культуры здоровья», «трудовое»,
«экологическое», «ценности научного познания»). Группировка ФГОС не полностью
совпадает с 8 каноническими направлениями (в ФГОС «гражданское» и «патриотическое»
часто слиты в один блок) — при сборке каталога расщепляем:
гражданско-патриотический блок ФГОС → формулировки про гражданскую идентичность
идут в `Гражданское`, формулировки про любовь к Родине и символы — в `Патриотическое`.
Дублирование одной и той же формулировки в две ячейки допустимо.

**Объём.** 3 уровня × 8 направлений × 5–8 формулировок = ~120–192 строки.

**Сбор.** Ручной проход по PDF приказов перед реализацией задач плана (отдельный
подготовительный шаг, не задача исполнителя кода). Каждая ячейка комментируется
ссылкой на пункт стандарта.

**Маппинг 11 направлений на 8 канонических:**
```
'Гражданское'            → 'Гражданское'
'Патриотическое'         → 'Патриотическое'
'Духовно-нравственное'   → 'Духовно-нравственное'
'Эстетическое'           → 'Эстетическое'
'Физическое и здоровье'  → 'Физическое и здоровье'
'Трудовое'               → 'Трудовое'
'Экологическое'          → 'Экологическое'
'Познавательное'         → 'Познавательное'   // ценность научного познания
'Семейные ценности'      → 'Духовно-нравственное'
'Профориентация'         → 'Трудовое'
'Здоровый образ жизни'   → 'Физическое и здоровье'
```

Точные UI-лейблы 11 направлений остаются как сейчас в `DIRECTIONS`. Маппинг
действует только для выбора ячейки каталога.

## 5. Промпт — секция каталога

В `buildSkeletonMessages`, после `[GOOD_EXAMPLES]`:

```
[PERSONAL_RESULTS_CATALOG]
Список планируемых личностных результатов из ФГОС {уровень},
направление «{каноническое направление}»:

1. {формулировка 1}
2. {формулировка 2}
...
8. {формулировка 8}

Выбери 3–5 формулировок из списка выше, наиболее релевантных теме «{topic}».
Верни их ДОСЛОВНО, без правок и сокращений, в поле "personalResults" каркаса.
Не придумывай свои формулировки — только из этого списка.
[/PERSONAL_RESULTS_CATALOG]
```

В `skeletonSchema` поле `personalResults` остаётся `optional` — если модель не
вернула или вернула невалидное, в `stream.ts` сработает `selectPersonalResults`
с fallback'ом.

## 6. Whitelist-валидация на выходе

```ts
// lib/scenario/personal-results.ts
export function selectPersonalResults(
  items: string[] | undefined,
  catalog: string[]
): string[] {
  const norm = (s: string) => s.replace(/\s+/g, ' ').trim()
  const catalogNormed = new Set(catalog.map(norm))
  const valid = (items ?? [])
    .map(norm)
    .filter(s => catalogNormed.has(s))
  const deduped = Array.from(new Set(valid))
  if (deduped.length >= 3) return deduped.slice(0, 5)
  // fallback: добираем недостающее из каталога, сохраняя порядок
  const need = 3 - deduped.length
  const fallback = catalog
    .map(norm)
    .filter(s => !deduped.includes(s))
    .slice(0, need)
  return [...deduped, ...fallback]
}
```

Гарантии:
- В `ScenarioContent.personalResults` НИКОГДА не попадает формулировка вне каталога.
- При пустом/невалидном ответе LLM — минимум 3 канонические формулировки из ФГОС
  (по умолчанию первые из каталога ячейки — это значит, что у методиста всегда
  есть «гарантированный минимум»).
- На сохранении пользовательских правок в редакторе whitelist НЕ применяется
  (учитель имеет право подправить формулировку под класс — это его сценарий,
  не корпус и не публикация в shared).

## 7. Возрастной кап длительности

### 7.1. Сервер — `generationInputSchema`

```ts
.superRefine((data, ctx) => {
  const cap = data.grade === 1 ? 35 : 45
  if (data.durationMin > cap) {
    ctx.addIssue({
      code: 'custom',
      path: ['durationMin'],
      message: data.grade === 1
        ? 'Для 1 класса длительность не более 35 мин (СанПиН).'
        : `Для ${formatGrade(data.grade)} длительность не более 45 мин (СанПиН).`,
    })
  }
})
```

Срабатывает на сервере во ВСЕХ путях: `/api/generate/stream`, server actions,
вручную составленный запрос. Возвращается 400 с понятным сообщением.

### 7.2. UI — `/app/new`

Чисто клиентский UX:
```
const cap = grade === 1 ? 35 : 45
const allowed = DURATIONS.filter(d => d <= cap)
```
При переключении класса 60-минутная опция исчезает для 1 класса. Существующий
дефолт (30 мин) допустим для всех. Сервер всё равно валидирует.

### 7.3. На что НЕ распространяется

- Существующие сценарии в БД (если кто-то когда-то создал 60-мин для 1 класса —
  они открываются и экспортируются как есть).
- Регенерация активности (`regenerateActivityAction`) — там не передаётся
  длительность занятия целиком, только активность.

## 8. Проверка наличия рефлексии

В `lib/scenario/quality.ts` — расширяется существующая `checkScenario`,
возвращающая `qualityWarnings: string[]`:

```ts
// нет рефлексии вообще
if (content.stages.every(s => s.kind !== 'reflection')) {
  warnings.push('В сценарии нет этапа рефлексии (заключительная часть)')
}

// рефлексия без вопросов
for (const stage of content.stages) {
  if (stage.kind !== 'reflection') continue
  const hasQuestions = stage.activities.some(a =>
    (a.questions?.length ?? 0) > 0 || a.text.includes('?')
  )
  if (!hasQuestions) {
    warnings.push('В этапе рефлексии нет вопросов для обратной связи')
  }
}
```

Warnings уже отображаются неблокирующим баннером в редакторе (как PII-warning).
Новых UI-каналов не вводится.

## 9. Методическая шапка экспорта

Текущая `metaTable` в `lib/export/document-model.ts` выводит 4 поля. Расширяем
до полноценного методического бланка (порядок строго фиксированный):

| Поле в шапке | Источник |
|---|---|
| Тема | `meta.topic` (если пусто — «—») |
| Направление воспитания | `direction` (полный лейбл из DIRECTIONS) |
| Класс / уровень | `formatGrade(grade) + ' (' + levelLabel(level) + ')'` |
| Длительность | `${meta.durationMin} мин` |
| Формат | `meta.format` |
| Цель занятия | `content.goals[0]` + ` (и др.)` при `length > 1` |
| Формируемые ценности | `content.values?.join(', ')` (строка опускается, если пусто) |
| Оборудование | `content.materials.join(', ')` |

Под шапкой — заголовок **«Планируемые личностные результаты»** и буллетный список
из `content.personalResults`. Если массив пуст — заголовок и список не рендерятся.

Ход занятия (этапы) — как сейчас. Дисклеймер ИИ в футере и промо-PNG не трогаем.

## 10. Совместимость

- Поле `personalResults` — `optional` в zod → старые сценарии валидируются.
- Карточка редактора скрывает кнопку «Сгенерировать заново», ничего не предлагает
  автоматически — учитель может оставить блок пустым.
- В шапке экспорта пустые ячейки выводятся как «—».
- Возрастной кап применяется только на ВХОДЕ генерации; старые длинные сценарии
  не затрагиваются.

     не падают; шапка с «—» в новых ячейках; карточка результатов пуста.
  3. Форма `/app/new`: класс=1 → опция 60 мин недоступна; попытка POST через
     `curl` с `durationMin=45` → 400 «Для 1 класса…».
  4. Сценарий без `reflection`-этапа (отредактировать вручную и сохранить) →
     баннер с warning «В сценарии нет этапа рефлексии».

## 13. Деплой

- Миграций нет — `git pull && docker compose up -d --build`.
- После деплоя опционально (вручную):
  - перегенерить библиотеку сообщества под новый скелет:
    `pnpm exec tsx scripts/seed-shared.ts --force` против прод-БД
    (старые shared-сценарии работоспособны, но без блока личностных результатов).
- Changelog запись **v1.10.0** (feature) видна на `/changelog`.
