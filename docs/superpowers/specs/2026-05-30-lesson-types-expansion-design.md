# Расширение типов занятий: РоВ + 4 группы внеурочного поля

**Дата:** 2026-05-30
**Статус:** Утверждена в брейншторме, требует ручного UAT после реализации.
**Опорные документы:**
- Аналитический обзор «Классный час и Разговоры о важном: нормативные требования ФГОС и ФОП» (в проекте, нормативная база приказы 286/287/413/372/370/371, Указ 809, Письмо Минпросвещения 06-221).
- `docs/fgos-fop-compliance.md` (внутренний разбор, был основой для P1+P2 фазы личностных результатов).
- Существующая спека `docs/superpowers/specs/2026-05-30-fgos-fop-compliance-p1-p2-design.md` (личностные результаты, возрастной кап, рефлексия).

## 1. Цели

1. Снять монокультуру «РоВ» с пайплайна генерации: в `/app/new` учитель явно выбирает один из 5 типов занятия, и форма + промпт + валидаторы заточены под выбранный тип.
2. Закрыть «пустой лист» там, где у учителя сейчас НЕТ готовых федеральных сценариев: тематический кружок, функциональная грамотность, предметное углубление, воспитательное мероприятие.
3. Сохранить уже работающую РоВ-стилистику (трёхчастная структура, личностные результаты ФГОС, видеовход) как одну из 5 веток, без регрессий для существующих сценариев.
4. Подготовить модель данных и промпты к появлению собственных RAG-корпусов под каждый тип (ПИЗА для функграма, профильные пособия для предметки и т.д.) — в этой итерации это backlog, но структура готова.

## 2. Не-цели

- Свои RAG-корпуса для функграма/кружка/предметки — за рамками.
- Whitelist метапредметных и предметных результатов по ФГОС-каталогу — за рамками (свободный ввод).
- Подтипы функграма как отдельные UI-поля — есть `literacyKind` (4 значения), детализация откладывается.
- Профориентация как отдельный тип — это направление внутри РоВ/воспит.мероприятия (см. §3).
- Калибровка качественных порогов для не-РоВ типов — отдельная фаза после первого живого UAT.
- Своя возрастная группировка РоВ как поле формы — учитель выбирает класс, группа вычисляется в промпте.
- Изменения формата экспорта PDF/DOCX, кроме лейбла главного классификатора.

## 3. Финальный набор типов (5)

Идентификатор `lesson_type`, значения:

1. `rov` — «Разговоры о важном». Жёсткая трёхчастная структура, обязательный видеовход в мотивационной части, whitelist личностных результатов ФГОС, направление воспитания обязательно.
2. `krujok` — тематический кружок / занятие по интересам. Свободная форма, без жёсткой структуры, без видеовхода, без whitelist личностных результатов.
3. `literacy` — функциональная грамотность. Структура «жизненный кейс → разбор → перенос на новый кейс». Обязательное поле `literacyKind` ∈ {`reading`, `math`, `financial`, `science`}.
4. `subject_extension` — предметное углубление (опыт, проект, олимпиадная подготовка, лаборатория поверх школьного предмета). Обязательное поле `subject` (свободный ввод: «Физика», «Биология»…). Вместо направления воспитания — предмет.
5. `event` — воспитательное мероприятие / тематический классный час (не РоВ), праздник, КТД, тематический день. Мягкая структура, направление воспитания обязательно, личностные результаты опциональны (но whitelist применяется, если направление выбрано).

Профориентация (`Россия — мои горизонты`, `Билет в будущее`) — НЕ отдельный тип. Это направление воспитания внутри `rov` и `event` (уже есть в `DIRECTIONS`).

## 4. Соответствие нормативке (по типам)

Источник — аналитический обзор §3, §4, §5.

| Параметр | rov | krujok | literacy | subject_extension | event |
|---|---|---|---|---|---|
| Главное поле классификации | направление воспитания | свободная тематика (в `topic`) | вид грамотности | школьный предмет | направление воспитания |
| Структура этапов | жёсткая трёхчастная (мотив. → основная → итог) | свободная | кейс → разбор → перенос | гипотеза/задача → проверка/эксперимент → вывод/обобщение | мягкая трёхчастная без обязательного видео |
| Видео в мотивационной части | обязательно | опц. | опц. | опц. | опц. |
| Личностные результаты ФГОС | обязательны 3–5, whitelist по каталогу | опц., без whitelist | опц., без whitelist | опц., без whitelist | опц., whitelist если направление задано |
| Метапредметные результаты (УУД) | опц. (free, без каталога) | опц. | опц. | опц. | опц. |
| Предметные результаты | опц. | опц. | опц. (но желательны) | опц. (но желательны) | опц. |
| Возрастные группы | РоВ: 1–2/3–4/5–7/8–9/10–11 (вычисляется из `grade` через `gradeToRovGroup`, попадает в промпт) | произвольный класс | произвольный класс | произвольный класс | произвольный класс |
| СанПиН-кап длительности | как сейчас (1 класс → 35 мин, остальные → 45) | как сейчас | как сейчас | как сейчас | как сейчас |

Дополнительные нормативные требования, явно вшиваемые в `rov`-промпт (упущенные до этой итерации):
- Обсуждение видеоматериала — первое, что делают на занятии (первый блок мотивационной части).
- «Точное усвоение нового термина не является целью» (§4.5 обзора) — не педалировать заучивание определений.
- Адаптация под региональный/этнокультурный контекст и состав семей класса (§4.5) — упоминание в промпте + подсказка учителю в UI.
- Сочетание 5 видов деятельности в основной части (§4.3): интеллектуальной, коммуникативной, практической, игровой, творческой. Реализуется упоминанием в промпте — наша типизация блоков (`engage`/`main`/`discussion`/...) не пересобирается под нормативные виды деятельности (это слишком инвазивно).

Дополнительные требования к ВД в целом (§4.6 обзора) — для всех типов:
- Триада результатов (личностные + метапредметные + предметные) — теперь все три поля есть в `ScenarioContent`, опц. везде кроме `rov.personalResults`.
- Отметки не выставляются — это уже не противоречит нашему пайплайну.

## 5. Модель данных

### 5.1. Миграция 0014

```sql
ALTER TABLE scenarios ADD COLUMN lesson_type text NOT NULL DEFAULT 'rov';
ALTER TABLE scenarios ADD CONSTRAINT scenarios_lesson_type_check
  CHECK (lesson_type IN ('rov','krujok','literacy','subject_extension','event'));

ALTER TABLE shared_scenarios ADD COLUMN lesson_type text NOT NULL DEFAULT 'rov';
ALTER TABLE shared_scenarios ADD CONSTRAINT shared_scenarios_lesson_type_check
  CHECK (lesson_type IN ('rov','krujok','literacy','subject_extension','event'));
CREATE INDEX shared_scenarios_lesson_type_idx ON shared_scenarios (lesson_type);

ALTER TABLE rag_documents ADD COLUMN lesson_type text;
-- nullable: РоВ-корпус и seed помечаются в backfill-скрипте; новые типы добавятся, когда появятся корпуса.
```

Backfill:
- Существующие `scenarios` и `shared_scenarios` получают `'rov'` через `DEFAULT` (всё сгенерировано РоВ-промптом).
- `rag_documents`: отдельный одноразовый `UPDATE rag_documents SET lesson_type='rov' WHERE source IN ('razgovor','seed')`. Это нужно сделать в `scripts/backfill-rag-lesson-type.ts` (идемпотентно).

`scenarios.direction` остаётся `text NOT NULL` для обратной совместимости. Для `krujok`/`literacy`/`subject_extension` пишем пустую строку или строковое представление подтипа (см. §5.3). Альтернатива «сделать nullable» отклонена — много мест читают, поломки гарантированы.

### 5.2. `ScenarioContent` — новые опц. поля

```ts
// lib/scenario/schema.ts
scenarioContentSchema = z.object({
  // ... существующее
  personalResults: z.array(z.string().min(1)).max(8).optional(),
  metaResults: z.array(z.string().min(1)).max(10).optional(),     // НОВОЕ — УУД
  subjectResults: z.array(z.string().min(1)).max(10).optional(),  // НОВОЕ — предметные
  subject: z.string().min(1).max(80).optional(),                  // НОВОЕ — для subject_extension
  literacyKind: z.enum(['reading','math','financial','science']).optional(), // НОВОЕ — для literacy
});
```

Skeleton-schema — соответствующие поля как `.optional()` без жёсткой валидации (LLM может прислать мусор, отфильтруется на финальной проверке).

### 5.3. `generationInputSchema` — ветвление через `superRefine`

```ts
generationInputSchema = z.object({
  lessonType: z.enum(['rov','krujok','literacy','subject_extension','event']),
  topic: z.string().min(3),
  grade: z.number().int().min(1).max(12),
  durationMin: z.number().int(),
  format: z.string(),
  direction: z.string().optional(),         // обязательно для rov/event
  subject: z.string().optional(),           // обязательно для subject_extension
  literacyKind: z.enum([...]).optional(),   // обязательно для literacy
  // ... остальные поля как были
}).superRefine((v, ctx) => {
  // существующий СанПиН-кап
  // НОВОЕ:
  if ((v.lessonType === 'rov' || v.lessonType === 'event') && !v.direction) {
    ctx.addIssue({path:['direction'], message:'Выберите направление воспитания'});
  }
  if (v.lessonType === 'subject_extension' && !v.subject) {
    ctx.addIssue({path:['subject'], message:'Укажите школьный предмет'});
  }
  if (v.lessonType === 'literacy' && !v.literacyKind) {
    ctx.addIssue({path:['literacyKind'], message:'Выберите вид функциональной грамотности'});
  }
});
```

Запись в БД: `scenarios.direction` для `subject_extension` = значение `subject` (для админ-статистики удобнее видеть «Физика» в той же колонке); для `literacy` = label `literacyKind` («Читательская грамотность»…); для `krujok` = `'—'` или пусто, осознанно. Это решение жертвует семантической чистотой колонки ради того, чтобы существующая админ-статистика (`barList` по `direction`) продолжала рендериться без падений. Альтернатива — рефакторить `lib/admin/stats.ts` под per-тип агрегаты — выходит за scope.

### 5.4. Справочники

`lib/scenario/options.ts`:

- `LESSON_TYPES: {value, label, description, icon}[]` — массив из 5 элементов для wizard step 1.
- `DIRECTIONS` — добавить 9-е направление **«Адаптация к изменяющимся условиям социальной и природной среды»** (для ООО/СОО). Для НОО оно не предусмотрено — UI фильтрует список направлений в зависимости от уровня (учитель 1–4 класса его не видит).
- `LITERACY_KINDS = [{value:'reading', label:'Читательская грамотность'}, ...]` — 4 значения.
- `FORMATS` — расширить новыми форматами для не-РоВ типов:
  - kружок: + «Мастер-класс», «Творческая мастерская».
  - literacy: + «Практикум», «Кейс-сессия» (игра уже есть).
  - subject_extension: + «Лабораторная», «Эксперимент», «Проект», «Олимпиадный тренинг».
  - event: + «Праздник», «Тематический день», «КТД».
  Старые 8 форматов остаются доступны для любого типа.
- `DURATIONS` — расширить до `[20, 30, 40, 45, 60, 90]`. РоВ ограничен СанПиН-капом (20/30/40 для 1 класса, 20/30/40/45 для остальных в зависимости от существующей логики). Не-РоВ типы — без жёсткого верхнего ограничения свыше СанПиН.

### 5.5. Каталог личностных результатов

`lib/scenario/personal-results.ts` (`CATALOG`):
- Добавить 9-е направление `Адаптация` для ООО/СОО — 2–4 дословные формулировки из ФГОС ООО приказ 287 п. 42, ФГОС СОО приказ 413 п. 7.1. Точные формулировки взять из обзора §4.6 («адаптация к изменяющимся условиям социальной и природной среды»).

### 5.6. Возрастные группы РоВ

`lib/scenario/levels.ts` — новая функция `gradeToRovGroup(grade: number): '1-2' | '3-4' | '5-7' | '8-9' | '10-11' | 'СПО'`. Используется ТОЛЬКО в РоВ-промпте (`buildRovPrompt`). Грейд формы остаётся `integer 1..12`.

## 6. UI: двухшаговый wizard `/app/new`

### 6.1. Шаг 1 — выбор типа

`app/app/new/page.tsx` рефакторится:
- Если query-параметр `?type=` отсутствует → рендер `<LessonTypePicker>`: 5 карточек в сетке (адаптивная: 1 / 2 / 3 колонки), у каждой иконка из `lucide-react`, заголовок, 1–2 строки описания, и ссылка `/app/new?type={value}&...` (другие query-параметры прокидываются — `topic`, `planTopicId`, `calendarDate`).
- На карточке РоВ — бейдж «Федеральный курс» (мелким шрифтом), для остальных — бейдж «Программа школы».
- Кнопка «Назад на дашборд».

### 6.2. Шаг 2 — форма под тип

При наличии `?type=` рендерится та же страница, что сейчас, но:
- Хедер: «Создать сценарий → {label типа}», ссылка-крошка «← Изменить тип» возвращает на шаг 1.
- Поля формы — фильтруются по типу:
  - `rov`, `event`: показывается `<select name="direction">` (как сейчас).
  - `subject_extension`: вместо направления — `<input name="subject" placeholder="Физика, Биология...">`.
  - `literacy`: вместо направления — `<select name="literacyKind">` (4 значения).
  - `krujok`: блок «Главное поле классификации» скрыт, только тема. Подсказка под полем темы: «Сформулируйте, чем занимается кружок: например, «Робототехника на Arduino: первое знакомство».»
- Список `<option>` форматов — фильтруется по типу (см. §5.4).
- Список `<option>` длительностей — фильтруется по СанПиН-капу (как сейчас) И по доступным для типа.
- Источник темы (`manual`/`calendar`/`plan`) — остаётся как есть, актуален для всех типов.
- Блок «Свой материал» — остаётся как есть.

### 6.3. Pre-match шаг

Двухшаговый flow «Подобрать похожие → Сгенерировать новый» сохраняется, но pre-match получает hard-фильтр `lesson_type = ?`. Если для выбранного типа в shared_scenarios пусто → шаг пропускается, сразу генерация (UI: тонкая подсказка «В библиотеке пока нет сценариев этого типа — генерируем новый»).

### 6.4. Передача `lessonType` в payload

`lessonType` приходит из query (`?type=`) и прокидывается в форму как hidden input. Поля `subject` / `literacyKind` — видимые контролы, рендерятся в форме под выбранный тип; на сервер уходят как обычные поля formData.

## 7. Промпты: разделение по типам

Отказались от A1 (один файл с веткой `if`) в пользу A2: **папка `lib/scenario/prompts/`**.

```
lib/scenario/prompts/
  shared.ts          # общие хелперы: JSON-формат, секции RELEVANT_METHODOLOGY, TEACHER_MATERIAL, GOOD_EXAMPLES, правило фактов, contextBlock
  rov.ts             # buildRovSkeletonMessages, buildRovBlockMessages
  krujok.ts          # build*KrujokMessages
  literacy.ts        # build*LiteracyMessages
  subject.ts         # build*SubjectExtensionMessages
  event.ts           # build*EventMessages
  index.ts           # buildSkeletonMessages(input, deps) — диспетчер по input.lessonType
```

`lib/scenario/prompt.ts` остаётся как тонкий barrel-реэкспорт `index.ts` ради обратной совместимости импортов в `gen-seed.ts`, `seed-shared.ts`, `regenerate.ts`, `stream.ts`. Все они переключаются на `index.ts` в этой же итерации; `prompt.ts` через 1–2 итерации удаляется.

Каждый модуль типа экспортирует:
- `buildSkeletonMessages(input, deps): ChatMessage[]`
- `buildBlockMessages(input, skeleton, block, context, deps): ChatMessage[]`
- `PROMPT_VERSION: string` (свой для типа, дискриминируется префиксом: `v1-rov-2026-05-30`, `v1-krujok-2026-05-30` и т.д.).

`shared.ts` экспортирует:
- `JSON_FORMAT_HINT` — общий формат ответа.
- `buildMethodologyBlock(chunks)`, `buildMaterialBlock(text)`, `buildGoodExamplesBlock(examples)` — переиспользуемые секции.
- `RULE_NO_HALLUCINATIONS` — правило фактов из §4.5 (с расширением «нет в TEACHER_MATERIAL ИЛИ методичках»).
- `RULE_NO_GRADING` — отметки не выставляются.

Диспетчер `index.ts`:

```ts
export function buildSkeletonMessages(input: GenerationInput, deps: ...): ChatMessage[] {
  switch (input.lessonType) {
    case 'rov': return buildRovSkeletonMessages(input, deps);
    case 'krujok': return buildKrujokSkeletonMessages(input, deps);
    // ...
  }
}
```

`PROMPT_VERSION` диспетчер: `getPromptVersion(lessonType)`. Пишется в `generations.prompt_version` (уже есть колонка).

### 7.1. РоВ-специфика (rov.ts)

В дополнение к существующему промпту v9 добавляется:
- Жёсткое требование: первый блок мотивационной части — обсуждение видеоматериала (учитель сам подберёт ролик; LLM не выдумывает ссылку, а описывает, что обсуждать).
- «Не педалируй заучивание терминов; через год к этой теме вернёмся».
- «Адаптируй задания под региональный/этнокультурный контекст и состав семей класса (учитель уточнит)».
- Сочетание 5 видов деятельности в основной части (упоминание).
- Возрастная группа РоВ (`gradeToRovGroup`) попадает в системное сообщение: «Сценарий для возрастной группы 5–7 классов курса „Разговоры о важном“».
- `PERSONAL_RESULTS_CATALOG` секция — как сейчас.

### 7.2. krujok.ts

- «Это занятие тематического кружка / клуба по интересам. Структура свободная, без обязательной трёхчастности.»
- «Не используй обязательный видеовход; включай видео/демо только если оно естественно для темы.»
- Главный результат — развитие интереса и/или конкретное практическое умение.
- Нет секции `PERSONAL_RESULTS_CATALOG`; `personalResults` опц. свободный список.

### 7.3. literacy.ts

- «Это занятие по {literacyKind: читательская/математическая/финансовая/естественнонаучная} грамотности. Опирайся на формат задач PISA/ФИОКО: реальный жизненный контекст → решение → перенос на новый контекст.»
- Структура блоков: «Введение контекста (короткая мотивация) → Кейс-задача → Разбор → Перенос на похожую задачу → Рефлексия умения».
- `subjectResults` — желательно: что конкретно научились делать (формулировки в терминах умений).
- Видеовход не требуется.

### 7.4. subject.ts

- «Это внеурочное занятие по предмету {subject} — углубление, опыт, проект или олимпиадная подготовка.»
- Структура: «Гипотеза/задача → План работы → Выполнение (эксперимент/исследование) → Обсуждение результата → Перенос/обобщение».
- `subjectResults` — желательно.

### 7.5. event.ts

- «Это воспитательное мероприятие — тематический классный час (не РоВ), праздник, КТД или тематический день, по программе воспитания школы.»
- Мягкая трёхчастная структура: мотивация → основная → итог. Видеовход — опц.
- Направление воспитания обязательно (валидируется в §5.3), поэтому whitelist `personalResults` применяется всегда — как в РоВ.

### 7.6. Embed-строка для pre-match и pre-match-RAG

`lib/community/prematch.ts` и `lib/rag/retrieve.ts` строят строку для эмбеддинга. Сейчас: `"{direction} {grade} {topic} {format}"`. Меняем на тип-зависимое:

```ts
function buildEmbedQuery(input): string {
  const parts = [input.topic, input.format];
  if (input.lessonType === 'subject_extension' && input.subject) parts.unshift(input.subject);
  if (input.lessonType === 'literacy' && input.literacyKind) parts.unshift(literacyLabel(input.literacyKind));
  if (input.direction) parts.unshift(input.direction);
  parts.push(`${input.grade} класс`);
  return parts.join(' ');
}
```

Этим достигается «поиск по предмету» через семантику (без отдельной колонки/индекса).

## 8. Качество (`lib/scenario/quality.ts`)

`checkBlock` и `checkScenario` ветвятся по `lessonType` через новый аргумент. Гейт-пороги (env-тюнятся):

**Общие для всех типов** (применяются всегда):
- `MIN_BLOCK_CHARS` (600 по умолчанию) — длина блока.
- Дубли названий этапов — warning.
- Покрытие `coreMeanings` — warning.
- Общий объём сценария ≥ `MIN_SCENARIO_CHARS` — warning.

**Только для `rov` и `event`** (РоВ-стилистика бесед):
- ≥2 реплик «Учитель:» в блоках `engage`/`main`.
- Каждая реплика «Учитель:» ≥`MIN_TEACHER_TURN_CHARS` (40).
- ≥3 вопросов в `discussion`-блоках.
- Каждый вопрос ≥`MIN_QUESTION_CHARS` (15) и содержит «?».
- Warning «нет этапа рефлексии».
- Warning «в рефлексии нет вопросов».

**Только для `krujok`, `literacy`, `subject_extension`** (практическая стилистика):
- Длина шага ≥`MIN_STEP_CHARS` (200) — мягче.
- Хотя бы один блок содержит действие учителя/ученика (эвристически: глаголы в повелительном наклонении или «учитель», «обучающиеся»). Сначала ставим простой эвристический детектор; калибруем после первого UAT.

Пороги для не-РоВ типов **намеренно намного мягче** — мы не знаем реальный объём качественного выхода для них. После первого UAT (отдельный backlog) калибруем.

`generateBlockWithGate` получает аргумент `lessonType` и передаёт в `checkBlock`.

## 9. Pre-match, библиотека, RAG

### 9.1. Pre-match

`lib/community/prematch.ts` — `findSimilarShared` принимает `lessonType` и применяет hard-фильтр `WHERE lesson_type = ?`. При нулевых результатах функция возвращает пустой список → UI пропускает шаг.

`useSharedAsIsAction` копирует shared-сценарий в `scenarios` с тем же `lesson_type` (берём из исходного `shared_scenarios.lesson_type`).

### 9.2. `/app/library`

Селект «Тип занятия» сверху страницы (5 значений + «Все»). Дефолт — «Все», чтобы новая библиотека не выглядела пустой. Карточка `SharedCard` получает бейдж типа в углу.

Семантический поиск по `q` работает по всем типам, фильтр накладывается поверх через `WHERE`.

### 9.3. RAG retrieve

`lib/rag/retrieve.ts` — пока **без** жёсткого фильтра по `rag_documents.lesson_type`. Все типы тянут из общего корпуса РоВ + seed. Это осознанный компромисс: своих корпусов для функграма/предметки нет, отбирать «только нелетящие» нечего. Когда корпус появится — добавляем `WHERE lesson_type IS NULL OR lesson_type = ?` (мягкий фильтр, как с направлением).

### 9.4. Pre-match-RAG для генерации

Существующая логика подмешивания shared как `GOOD_EXAMPLES` в промпт — тоже фильтруется по `lesson_type` (нет смысла подсовывать РоВ-сценарий как пример для кружка).

## 14. Открытые риски и решения

- **Качественные пороги для не-РоВ типов** — мы не знаем реальный объём качественного выхода. Решение: ставим мягкие пороги, помечаем backlog «калибровка после первого UAT».
- **Семантический поиск по предмету/виду грамотности** — мы полагаемся на cosine. Если на практике подмешивает «не свой» предмет — добавим soft-фильтр по `subject` через эвристику (substring match) поверх. Решение: мониторим, не делаем сразу.
- **`scenarios.direction` как полиморфная колонка** — жертвуем чистотой ради совместимости. Решение: задокументировано в §5.3; рефакторинг админ-статистики — backlog.
- **Wizard step 1 ломает прямые URL** — старые ссылки на `/app/new?topic=...` без `?type=` будут перебрасывать на шаг 1. Решение: это приемлемо, потому что таких ссылок снаружи нет (только из плана и календаря — оба сейчас идут на дашборд). При переходе из плана/календаря добавляем `?type=rov` по умолчанию (т.к. учитель шёл за «классическим» сценарием).
- **РоВ-промпт после переезда в `prompts/rov.ts` должен пройти регрессионный UAT** против эталона v9 — иначе риск ухудшения. Решение: имплементер обязан запустить живой UAT генерации одной РоВ-темы и сверить с baseline по объёму/качеству.
