# Дизайн — Мягкий фильтр направления в RAG-retrieve (#25)

**Дата:** 2026-05-24
**Статус:** утверждён
**Контекст:** backlog #25. У чанков «Разговоров о важном» (`source='razgovor'`, 510 шт.) `chunk_meta.direction = null` — в URL направление не размечено. Только 19 seed-чанков имеют направление.

## Проблема
`lib/rag/retrieve.ts` (`queryCandidatesLive`) применяет **жёсткий** фильтр направления: `AND chunk_meta->>'direction' = $direction`. Когда учитель задаёт направление, этот фильтр исключает все 510 РоВ-чанков (direction=null). РоВ-корпус возвращается только через fallback `if (rows.length < topK && direction) → повторный запрос с direction=null`. Но 6 из 11 направлений имеют ≥`topK` (3) seed-чанков (Патриотическое=4, Трудовое=4, Духовно-нравственное=3…), поэтому fallback НЕ срабатывает, и генерация для этих направлений опирается на 3-4 тонких seed-примера вместо богатых методичек РоВ. Это деградация качества retrieval для ~половины направлений.

(Проверено на живой БД 2026-05-24: `razgovor`=510 чанков direction=null; `seed`=19 чанков с направлением, по 1-4 на направление.)

## Решение (зафиксировано): мягкий фильтр — не исключать direction=null
Отклонены: классификация РоВ-чанков через GigaChat при ingest (дорого, точность под вопросом, всё равно нужен не-жёсткий фильтр); явный re-rank-бонус за направление (поднял бы слабые seed-чанки над методичками РоВ — нежелательно). Направление уже частично закодировано в эмбеддинге запроса (`"{direction} {topic}"`), а ранжирование cosine+BM25 само поднимает релевантные.

### Изменения в `lib/rag/retrieve.ts`
1. `queryCandidatesLive`: заменить жёсткий `dirFilter` на мягкий. Когда `direction` задан:
   ```
   AND (chunk_meta->>'direction' = $direction OR chunk_meta->>'direction' IS NULL)
   ```
   Когда `direction === null` — фильтра по направлению нет (как сейчас). Фильтр по грейду (`grade_min`/`grade_max`) и сортировка/лимит — без изменений.
2. `retrieveChunks`: убрать избыточный fallback-блок (`if (rows.length < topK && query.direction) { rows = await d.queryCandidates({ ...base, direction: null }) }`) — теперь null-чанки уже в первой выборке. Остаётся один вызов `d.queryCandidates({ ...base, direction: query.direction })`.
3. Ранжирование (`rankAndDiversify`) и всё остальное — без изменений.

## Поток данных
`retrieveChunks(query)` → `queryCandidates` (cosine+BM25, грейд-фильтр, мягкий direction-фильтр) → `rankAndDiversify` (0.7·cosine+0.3·BM25, ≤`maxPerDoc`/doc, top `topK`) → `RetrievedChunk[]`. Контракт функций не меняется.

## Обработка ошибок
Без изменений. `retrieveChunks` по-прежнему может бросить, если `embed` не вернул вектор; вызывающий код (`generate.ts`/`stream.ts`) ловит сбой retrieval как non-fatal.

## Тестирование
- `tests/lib/rag/retrieve.test.ts` (инъекция `queryCandidates`/`embed`): при заданном направлении `queryCandidates` вызывается РОВНО ОДИН раз (нет fallback); чанки без направления участвуют в результате; грейд-логика и ранжирование сохранены. Если в текущем тесте есть кейс на срабатывание fallback — переписать под новое поведение (fallback удалён).
- SQL-проверка мягкого фильтра — юнит-тест на `queryCandidatesLive` затруднён (ходит в БД), поэтому покрываем логику `retrieveChunks` через инъекцию; корректность SQL подтверждается ручным live-UAT.
- Гейты: `pnpm test`, `tsc --noEmit`, `pnpm lint`, `pnpm build`.
- **Живой UAT:** `retrieveChunks({ direction: 'Патриотическое', grade: 4, topic: '...' })` против прод-БД → среди результатов есть razgovor-чанки (раньше исключались). Сравнить с направлением без seed-перекоса.

## Риски
- Мягкий фильтр расширяет набор кандидатов (всегда включает РоВ-корпус) — это и есть цель; на качество влияет положительно (методички вместо тонких seed). Лимит кандидатов `RAG_CANDIDATES` (24) и диверсификация не меняются.
- Удаление fallback меняет число вызовов `queryCandidates` (был возможен 2-й) — покрыто тестом.
- Без миграций, без сетевых/ingest-прогонов в CI.

## Вне scope
Классификация направления РоВ-чанков; явный re-rank по направлению; изменение весов гибридного поиска; pre-match.
