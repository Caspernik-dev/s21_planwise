# SPEC: РоВ — приветствие отдельным блоком + видеовход с RuTube-поиском

**Дата:** 2026-06-01 · **Ветка:** `feat/rov-greeting-video` · **Опорный документ:** `materials/TZ_scenario_RoV.md` (§4.1 «Этап 1 — Мотивационно-целевой»).

## Цель фазы
Закрыть две жалобы пользователя:
1. **Первый блок всегда видео** — нет места «педагогическому приветствию» («Здравствуйте, ребята! Сегодня поговорим о…»). Это не соответствует реальному ходу занятия и устаревшему правилу в промпте.
2. **Видео без рабочей ссылки** — модель явно запрещено выдумывать URL, но и поиска нет. Учителю остаётся самостоятельно вбивать запрос в RuTube.

Решение: РоВ-engage всегда два блока — `Приветствие и постановка темы` (discussion) → `Видеовход` (video) — и видео-блок несёт `videoSearchQuery` (3-5 ключевых слов), который рендерится как URL `https://rutube.ru/search/?query=…` + QR в экспорте.

## Out of scope
- Усиление промпта по ценностям (минимум 2 secondary + 2 формулировки) — отдельная фаза.
- Динамическое разрешение `videoSearchQuery` в реальный URL (искать в RuTube API).
- QR на лендинге / в библиотеке.
- Альтернативные крючки (цитата, образ) — оставляем фиксированный формат «приветствие → видео» по решению пользователя.

## Архитектурные решения

### Схема (`lib/scenario/schema.ts`)
- `activitySchema`: добавляем `videoSearchQuery?: z.string().min(1).max(120).optional()`. Поле имеет смысл только для `type === 'video'`, но zod-схема разрешает на любом типе (валидатор не отбраковывает посторонние строки — это упрощает миграцию и не мешает экспорту).
- `Activity` тип расширяется автоматически через `z.infer`.

Без миграции БД (jsonb).

### Промпт (`lib/scenario/prompts/rov.ts`)
`PROMPT_VERSION → v13-rov-greeting-video-2026-06-01`.

**Skeleton-инструкции** (`buildRovSkeletonMessages`):
- Удалить строку «Первый блок мотивационной части — просмотр и обсуждение короткого видеоролика по теме» (старое жёсткое правило).
- Заменить на: «Мотивационно-целевой этап содержит РОВНО 2 блока: (1) `{type:"discussion", focus:"короткое приветствие учителя + постановка темы и цели занятия"}`, (2) `{type:"video", focus:"видеовход по теме — поисковой запрос для RuTube"}`. Не объединяй эти блоки в один.»
- В `SKELETON_SCHEMA_HINT` (JSON-shape) ничего не меняется на уровне stage-блоков; правило про 2 блока engage идёт текстом.

**Block-инструкции** (`buildRovBlockMessages`):
- Когда блок `type === 'video'` — добавить специфическую инструкцию:
  > Видеовход для РоВ — это просмотр короткого ролика (2-4 мин) по теме с последующим обсуждением. Не выдумывай конкретное название ролика и не давай прямых ссылок (`https://rutube.ru/video/...` или `youtube.com/watch...`) — их не существует. Вместо этого:
  > - в `text` опиши, КАК учитель строит работу с видео: ввод (1-2 фразы перед просмотром), показ ролика (одна строка-маркер `[Просмотр ролика]`), обсуждение по `questions`;
  > - в поле `videoSearchQuery` верни 3-5 ключевых слов на русском для поиска подходящего видеоролика на RuTube. Пример: `"День народного единства школьники"` или `"Дружба взаимопомощь мультфильм"`. Без кавычек, без длинных описаний, без URL.
- Когда блок engage[0] (kind === 'engage' и focus содержит «приветствие») — добавить:
  > Это первый блок занятия. Начни с короткого приветствия учителя (1-2 фразы: «Здравствуйте, ребята! Сегодня мы поговорим о…»), потом ОДНА содержательная реплика, формулирующая тему и цель. Не лей воды — приветствие занимает 1-2 минуты, потом сразу к делу.
- В `BLOCK_SCHEMA_HINT` дописать опц. поле:
  ```
  "videoSearchQuery"?: string,   // только для type:"video": 3-5 ключевых слов
  ```

### Серверная санитизация (`lib/scenario/rutube.ts` — НОВЫЙ файл)
Чистая логика, TDD:

- `RUTUBE_VIDEO_RE` / `YOUTUBE_RE` — regex для прямых ссылок (с http/https, опционально querystring).
- `sanitizeRutubeText(text: string): string` — удаляет / маскирует прямые ссылки на ролики. Заменяет `https://rutube.ru/video/abc123/` на `[Просмотр ролика]`. Не трогает search-URL вида `https://rutube.ru/search/?query=...` (они безопасны).
- `buildSearchUrl(query: string): string` — `https://rutube.ru/search/?query=${encodeURIComponent(query.trim())}`.
- `fallbackSearchQuery(topic: string, direction: string | undefined, leadingValue: string | undefined): string` — собирает запрос из темы+направления+ведущей ценности, обрезка до ~80 символов; используется, когда модель не дала `videoSearchQuery` или дала пусто.
- `extractOrFallbackQuery(activity, ctx): string | undefined` — оркестратор: если у активности `videoSearchQuery` валидный (не пустой, не URL) → берём; иначе пытаемся вытянуть подсказку из `text` (первые 5 слов после маркера «[Просмотр ролика]»); иначе fallback.

### Интеграция в стрим (`lib/scenario/stream.ts`)
После `parseBlock` для video-блоков — пропустить через санитизацию:
- `text` через `sanitizeRutubeText`;
- `videoSearchQuery` валидируется/добирается через `extractOrFallbackQuery`.

Чистая логика инъектируется через `lib/scenario/rutube.ts`, тесты не ходят в сеть.

### QR (`lib/export/qr.ts` — НОВЫЙ файл)
- `qrcode` v1.5.x переезжает из `devDependencies` в `dependencies` (~50 КБ, без нативных биндингов).
- `renderQrDataUrl(text: string, size = 160): Promise<string>` — обёртка над `QRCode.toDataURL(text, {margin: 1, width: size, color: {dark:'#0e4f30', light:'#ffffff'}})` (фирменный brand-700 dark). Возвращает `data:image/png;base64,...`.
- Lazy: вызывается только из путей экспорта PDF/DOCX. Юнит-тест: смоук на формат `data:image/png;base64,` префикса (без визуальной проверки).

### Экспорт PDF/DOCX (`lib/export/document-model.ts`, `to-pdf.tsx`, `to-docx.ts`)

**`document-model.ts`**: при сборке блока активности типа `video` (или любого, у кого есть `videoSearchQuery`):
- стандартная paragraph с `${label}. ${text}`;
- bullets вопросов как сейчас;
- НОВЫЙ тип блока `videoLink`: `{ type: 'videoLink'; query: string; url: string }` — добавляется ПОСЛЕ обычного paragraph активности.

**`to-pdf.tsx`**: рендер `videoLink` блока — горизонтальная карточка (brand-50 фон, brand-100 рамка): слева текст «🔍 RuTube: {query}» + ссылка-URL мелким моноширинным, справа `<Image src={qrDataUrl}>` 80×80 пт. URL — `<Link src={url}>` для кликабельности в PDF.

**`to-docx.ts`**: параграф «🔍 RuTube: {query}» + кликабельный hyperlink на URL отдельной строкой + `ImageRun` с PNG (160×160) на следующем абзаце. Использовать существующий паттерн `ImageRun` из docx v9.

### Редактор (`app/app/scenarios/[id]/editor.tsx`)
В рендере активностей: когда `activity.type === 'video'`, под существующими полями `text`/`questions` добавить:
- `<Input>` с label «Поисковой запрос на RuTube», bound to `activity.videoSearchQuery`.
- `<a href="https://rutube.ru/search/?query=...">` кнопка-ссылка «🔍 Открыть на RuTube» (target=_blank, rel=noopener); disabled-стиль если query пустой.

### Презентационный режим (`lib/scenario/slides.ts`)
Слайд для video-активности: после `text` или `questions` — отдельный буллет «RuTube: {query}» (без QR, чтобы не перегружать слайд). URL в полноэкранке не нужен — учителю достаточно знать запрос; для распечатки уже есть QR.
