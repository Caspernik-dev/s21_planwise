# Дизайн — Усиление детерминированного гейта качества (#26)

**Дата:** 2026-05-24
**Статус:** утверждён
**Контекст:** backlog #26. `lib/scenario/quality.ts` `checkBlock` проверяет объём/число реплик/число вопросов поверхностно: реплика «Учитель:» может быть пустой (считается, но не оценивается), вопросы только подсчитываются (`«?»` или `«что думаете?»` проходят). Цель — закрыть эти дыры без LLM-судьи.

## Решение (зафиксировано): усилить детерминированные проверки, БЕЗ LLM
Отклонён LLM-судья: прототип показал ненадёжность самопроверки модели и игнор инструкций; RAM/стоимость ограничены. Добавляем две объективные проверки в `checkBlock` (чистая функция, без сети).

### Изменения в `lib/scenario/quality.ts`, `checkBlock`
Новые env-пороги:
- `MIN_TEACHER_TURN_CHARS` (дефолт 40),
- `MIN_QUESTION_CHARS` (дефолт 15).

1. **Содержательность реплик «Учитель:».** Для `stageKind === 'engage' | 'main'`, когда реплики есть: разбить `text` по `/Учитель\s*:/`, отбросить преамбулу (элемент `[0]` до первого маркера), у каждой оставшейся реплики проверить `trim().length >= MIN_TEACHER_TURN_CHARS`. Любая короче → reason `«пустая или слишком короткая реплика «Учитель:»»`. (Существующая проверка «≥2 реплик» сохраняется.)
2. **Качество вопросов.** Для `block.type === 'discussion'`: помимо `questions.length >= 3`, каждый вопрос должен `include('?')` И `trim().length >= MIN_QUESTION_CHARS`. Любой не проходит → reason `«слишком короткий или неполный вопрос»`.

Существующие проверки (`MIN_BLOCK_CHARS` ≥600, ≥2 реплик для engage/main, ≥3 вопроса для discussion) — без изменений. Сигнатура `checkBlock(block, stageKind)` НЕ меняется → применяется и в генерации, и в регенерации через общий `generateBlockWithGate` автоматически.

`checkScenario` — без изменений.

## Поток данных
`generateBlockWithGate` (в `block-gen.ts`) вызывает `checkBlock`; при провале — заострённый ретрай до `MAX_BLOCK_RETRIES`, затем берётся лучшая попытка (graceful degradation, как сейчас). Более строгий гейт = возможны доп. ретраи, но не жёсткий отказ. Контракт не меняется.

## Калибровка
Пороги намеренно низкие — ловят только явный мусор, не нормальный РоВ-вывод: реплики «Учитель:» в РоВ — 3-6 предложений (~150-400 симв.), порог 40 отсекает лишь пустые/обрывочные; вопросы РоВ развёрнутые (>15 симв.), порог 15 отсекает `«?»`/`«как?»`. Оба env-тюнятся без передеплоя.

## Тестирование
`tests/lib/scenario/quality.test.ts` — добавить кейсы:
- блок engage/main с пустой репликой «Учитель:» при достаточной общей длине (≥600) и ≥2 маркерах → `ok:false`, reason про реплику;
- discussion с 3 вопросами, один из которых `«?»` (или короче 15 симв.) → `ok:false`, reason про вопрос;
- плотный корректный блок (длинные реплики, 3 полноценных вопроса) → `ok:true` (регресс — не сломать прошлый зелёный кейс);
- reflection не требует проверки реплик (как и раньше).
Гейты: `pnpm test`, `tsc --noEmit`, `pnpm lint`, `pnpm build`.

## Вне scope (сознательно)
- Обязательность маркера «Ответы обучающихся.» — легитимный `task`/`video`-блок может её не иметь (риск ложных отказов).
- Анти-повтор внутри блока — кросс-блочные повторы уже держит катящийся контекст (`buildRunningContext`).
- LLM-судья / семантическая оценка.

## Риски
- Более строгий гейт может вызвать доп. ретраи на реальном выводе — приемлемо (приоритет качества; деградация мягкая — берём лучшую попытку). Пороги низкие и env-тюнятся, чтобы не отбраковывать хороший РоВ-контент.
- Без миграций.
